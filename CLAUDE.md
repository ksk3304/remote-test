# CLAUDE.md - NicheScout プロジェクト

## プロジェクト概要

個人開発向けiOSアプリのニッチリサーチプロジェクト。

### 利用可能なスキル

| スキル | 用途 |
|--------|------|
| `/niche-scout` | AIが自動で狙い目ニッチを探索 |
| `/idea-check` | ユーザーのアイデアが狙い目かを検証 |

## ディレクトリ構成

```
output/
├── niche-scout/           # 従来型リサーチ（AIが自動探索）
│   └── niche-scout-report-vN.md
└── idea-check/            # アイデア検証型リサーチ
    └── [アプリ名]/
        └── report.md
```

### 注意：既存ファイルの扱い

`output/` 直下にある既存のレポート（`niche-scout-report.md` 等）は、移行前のレガシーファイル。
新規作成時は上記ディレクトリ構成に従うこと。

## ファイル命名規則

### /niche-scout（従来型）
- `output/niche-scout/niche-scout-report-vN.md` 形式（Nは連番）

### /idea-check（アイデア検証型）
- `output/idea-check/[アプリ名]/report.md` 形式
- アプリ名は英語小文字ケバブケース（例: `guru-tweet`）

## 調査ルール

### ジャンル選定について

**ユーザーにジャンルを聞かない。AIが自分で探す。**

- ジャンルは何でもいい。重要なのは75点以上のニッチを見つけること
- AIが市場調査を行い、有望なジャンルを自ら選定する
- ユーザーへの確認なしで調査を開始し、結果だけを報告する

### 新規調査時の必須事項

1. **必ず新しいファイルを作成する**
   - 既存レポートの最大バージョン番号 + 1 で新規作成
   - **既存ファイルを絶対に上書き・編集しない**
   - 過去の調査結果はそのまま残す

2. **過去調査ジャンルを避ける**
   - 調査前に `output/` 内の既存レポートを確認
   - 既に調査済みのジャンルは除外して新規ジャンルを調査

3. **レポート作成後はGitHubにプッシュする**
   - レポートファイルを作成したら、必ずGitHubにコミット＆プッシュする
   - コミットメッセージ例: `Add niche-scout-report-vN.md`

## レポート作成のガイドライン

1. **必須セクション**:
   - エグゼクティブサマリー（3ニッチの一覧表）
   - 各ニッチの詳細（市場分析、競合分析、不満分析、課金モデル、スコア、MVP仕様）
   - 推奨アクション
   - 調査履歴

2. **スコアリング基準**:
   - 75点以上: Go（開発着手可能）
   - 75点未満: Kill（作るだけ無駄）
   - **75点以上のニッチが3つ揃うまでリサーチを継続すること**

## スコアリング詳細（厳守）

### 計算式

```
総合スコア = (需要×0.25) + (競争優位性×0.20) + (改善余地×0.20) + (差別化×0.20) + (実装容易性×0.15)
```

### 各指標の評価基準（0-100点）

| 指標 | 重み | 80点以上の条件 | 60点以下の条件 |
|------|------|----------------|----------------|
| 需要 | 25% | 競合5本以上＋レビュー総数1000件以上＋検索キーワード3つ以上明確 | 競合3本未満 or レビュー少数 |
| 競争優位性 | 20% | 上位競合が放置（1年以上未更新）or 明確な品質問題あり＋大手不在 | 大手が複数参入＋高品質 |
| 改善余地 | 20% | 同一不満が3アプリ以上で各3件以上反復（構造的問題） | 不満が個人的・散発的 |
| 差別化 | 20% | 10文字以内で説明可能＋競合にない明確な強み | 差別化ポイントが曖昧 |
| 実装容易性 | 15% | ローカル完結＋単機能＋2週間で確実にMVP可能 | サーバ必須 or 複雑機能 |

### Kill条件（1つでも該当したら強制Kill、スコア関係なし）

- 競合がほぼゼロ（需要不明）
- 低評価が「好み」「個別事情」ばかりで改善余地が薄い
- 直接課金の理由を作れない（課金ベネフィットが曖昧）
- 個人開発で再現できない重機能が必須（SNS/マッチング/複雑同期など）

### 75点の壁

75点以上を達成するには、**全指標で平均75点以上が必要**。
つまり、1つでも55点以下の指標があると、他でカバーしても75点に届かない。

**甘い評価は禁止。根拠なく高スコアをつけない。**

## 調査の反省点（v5調査より）

### 失敗パターン：やってはいけないこと

1. **「シンプル」を差別化軸にするな**
   - 「シンプルな○○アプリ」は2020年代前半にやり尽くされた
   - 既に高品質な「シンプル特化」アプリが各ジャンルに存在する
   - 例：シンプル家計簿、継続する技術、シンプル血圧手帳など

2. **ニッチを決めてから競合を調べるな（順序が逆）**
   - ❌「このジャンルいけそう」→ 調べたら★4.5+の高品質競合がいた
   - ⭕「放置されている競合」を先に探す → そのジャンルを深堀り

3. **構造的不満を確認せずに飛び込むな**
   - 調べてみたら不満が「好み」「バグ」レベルばかりだった
   - **先に「3アプリ×3件以上の同一不満」があるか確認してから調査開始**

4. **需要が明確なジャンルばかり狙うな**
   - 需要が明確 ＝ 既に良いアプリがある
   - 競合が放置されている隙間を探すべき

### 正しい調査手順

```
1. まず「放置されている競合アプリ」を探す（1年以上更新なし、★3.5以下など）
   ↓
2. その競合のレビューで「構造的不満」があるか確認
   ↓
3. 3アプリ以上で同じ不満が反復していれば、そのジャンルを深堀り
   ↓
4. 高品質な競合がいないか最終確認
   ↓
5. スコアリング
```

### 70点未満が出たら即撤退

- 競争優位性が50点以下 → そのジャンルは即終了
- 改善余地が50点以下 → そのジャンルは即終了
- 明らかに勝てない競合がいる → 調査を続けても無駄

## 調査の反省点（v6調査より）

### 追加の失敗パターン

5. **放置競合を見つけても、別の高品質競合がいる問題**
   - 放置された競合（★3.2の睡眠日誌）を発見
   - 構造的不満も確認できた（昼夜逆転未対応、データ消失）
   - **しかし別の高品質競合（★4.6のbondavi）が存在**
   - 結果：競争優位性が確保できずスコアが伸びない

6. **アップデート停止でも「定番」として確立されているアプリ**
   - リズムケアは8年間アップデート停止
   - しかし★4.6を維持し「定番」として確立
   - 「広告なし」「定期更新」だけでは差別化にならない

### 2026年のiOSアプリ市場の現実

v1〜v6で25ジャンル以上を調査した結果、80点以上のニッチは0個、75点以上は4個だった。

**市場の特徴:**
- App Store開始から18年、主要ジャンルは成熟期
- ほぼすべてのジャンルで★4.0以上のアプリが複数存在
- 放置アプリを見つけても、別の高品質競合がアクティブに更新中

**対応策:**
- 基準を80点から75点に緩和（v6調査の結果を踏まえて決定）
- 75点以上のニッチ（血圧記録78.5点、読書記録77点、習慣トラッカー75.5点、植物水やり75点）で開発着手可能

## レポート品質ルール（v6より追加）

### 絶対に省略してはいけない内容

**75点未満でKill判定でも、以下は必ず詳細に記載すること:**

1. **競合分析**: 各競合アプリのURL、開発元、強み、弱み（ユーザーレビューからの具体的な不満）
2. **不満タグ集計表**: 複数アプリ横断で不満を整理
3. **構造的不満の判定**: 「3アプリ×3件以上」の基準を満たすか明記
4. **スコア計算の詳細**: 各指標の点数と根拠
5. **Kill理由の明確化**: どのKill条件に該当したか
6. **MVP仕様（参考）**: 仮に開発するならどうなるか

**Kill判定だからといってレポートを省略する理由にはならない。**

## 海外市場調査ルール

### 調査対象

75点以上（Go判定）のニッチのみ、海外市場を追加調査する。
75点未満のニッチは日本市場の調査で終了（効率重視）。

### 対象市場

| 優先度 | 市場 | 備考 |
|--------|------|------|
| 必須 | US（アメリカ） | 最大市場 |
| 必須 | UK（イギリス） | 英語圏 |
| 必須 | AU（オーストラリア） | 英語圏 |
| 可能なら | ドイツ | 欧州主要市場 |
| 可能なら | フランス | 欧州主要市場 |
| 可能なら | 韓国 | アジア主要市場 |

### 調査内容

1. **競合状況**: 各市場でのApp Store競合アプリ（評価、レビュー数、更新日）
2. **需要**: 検索ボリューム、カテゴリランキング
3. **差別化可能性**: 日本発の強みが活かせるか
4. **ローカライズ要件**: 翻訳以外に必要な対応（単位、文化的差異、法規制）

### グローバル展開判定基準

| 判定 | 条件 |
|------|------|
| **Go** | 英語圏で放置/低品質競合あり + ローカライズが軽微 |
| **Wait** | 市場はあるが高品質競合が存在 → 日本版で実績を作ってから |
| **No** | 文化的に日本特有 or ローカライズコストが高すぎる |

## 調査終了条件

**毎回の調査（1レポート）で、新規に75点以上のニッチを3つ発見すること。**

- 過去のレポートで発見済みのニッチはカウントしない
- 「累計で3つ見つけた」は達成にならない
- **その回の調査で新規に3つ**が必須

以下のいずれかを満たしたら調査終了:

1. **成功条件**: その回の調査で新規に75点以上のニッチを3つ発見
2. **終了条件**: 20ジャンル以上を調査して75点以上が3つ揃わない場合
   - その場合、全体を通じての総評・反省・今後の方向性提案をレポートに含めること

## /idea-check（アイデア検証）ルール

### 概要

ユーザーが持っているアプリアイデアの市場適合性を検証するスキル。
`/niche-scout` と同じスコアリング基準を使用するが、調査の起点がユーザーのアイデアになる。

### 情報収集フェーズ（必須）

リサーチを開始する**前に**、ユーザーから以下の情報を収集すること：

**必須情報（4項目）**
1. アプリ名（仮でOK）
2. コンセプト（どんなアプリか、1-2文）
3. ターゲット（誰が使うか）
4. 解決する課題

**任意情報（分かっていれば）**
- 想定課金モデル
- 差別化ポイント
- 技術的制約
- 参考にしている競合

### 評価の原則

1. **忖度しない**: ユーザーのアイデアだからといって甘い評価をしない
2. **正直に伝える**: Killならその理由を詳細に説明する
3. **代替案を提示**: Killの場合、可能なら改善方向やピボット案を提案

### レポート保存

- 保存先: `output/idea-check/[アプリ名]/report.md`
- アプリ名は英語小文字ケバブケース
- 保存後、GitHubにコミット＆プッシュ
